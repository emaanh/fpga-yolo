# tiny_yolo: streaming-video CNN object detector on FPGA (Icarus simulation)

A Tiny-YOLO-style object detector implemented in synthesizable Verilog and
verified end-to-end in Icarus. Three conv blocks + a 1Ă—1 detection head
classify and localize shapes on a 32Ă—32 RGB frame; the design is wired up
for a "streaming" feed by replacing the file-loader in the testbench with
a real camera pipeline, but the demo here runs entirely in simulation.

Per grid cell (4Ă—4) the head predicts `(tx, ty, tw, th, obj, cls0, cls1)`.
Decoding + NMS happens in Python after the RTL dumps cell logits.

## Fixed-point format

All activations and weights are Q4.12 signed (1 sign + 3 int + 12 frac,
range â‰ˆ [âˆ’8, +8)). Internal MAC accumulators are 32 bits (Q8.24). The
final write back to Q4.12 goes through `saturate.v`, so wraparound never
silently corrupts the output.

LeakyReLU uses slope = 1/8 so the negative branch is a single
arithmetic shift (`leaky_relu.v`).

Sigmoid is a 1024-entry combinational LUT generated by
`python/gen_sigmoid_lut.py`. Index = top 10 bits of (input with sign
flipped), which maps the Q4.12 range [âˆ’8, +8) onto the LUT evenly.

## Memory map

Two ping-pong feature maps (`fmap_mem.v`, distributed-RAM style):

```
load (ext)  -> A
conv1: A->B   pool1: B->A
conv2: A->B   pool2: B->A
conv3: A->B   pool3: B->A
head:  A->B   sigmoid: B->A
ext read of A holds the final sigmoided cells (112 words)
```

Each fmap stores a CHW-planar tensor: address = `c * H * W + y * W + x`.

Weight ROMs are layer-private, initialised from `.mif` files via
`$readmemh` in `weight_rom.v`. Weight layout (OIHW flattened):
`addr = co * Cin * 9 + ci * 9 + ky * 3 + kx` for 3Ă—3,
`addr = co * Cin + ci` for the 1Ă—1 head.

## Why distributed RAM, not BRAM?

`fmap_mem` and `weight_rom` use *combinational* read so the conv FSM can
do one MAC per cycle. On a real FPGA you'd swap these for BRAM and add a
one-cycle pipeline register on `rdata`, then push a single bubble into
the address counter. The MAC datapath itself doesn't change.

## Build & run

Requires Icarus Verilog 11+, Python 3.9+, numpy, pillow. Optional:
PyTorch for training.

```bash
# from tiny_yolo/
./scripts/run_sim.sh                   # one frame, random weights (smoke test)
./scripts/run_sim.sh -f 3              # frame_0003
./scripts/run_sim.sh --regen-data 2000 # regenerate 2000 synthetic frames
./scripts/run_sim.sh --trained         # quantize data/model.pt and run
./scripts/run_sim.sh --vcd             # also dump build/tb_top.vcd
```

Outputs land in:
- `build/sim_frame_XXXX.log`   â€” cycle-by-cycle stage transitions
- `data/detections/cells_XXXX.txt` â€” 112 raw Q4.12 hex values from the head
- `data/detections/frame_XXXX.txt` â€” `<cls> <cx> <cy> <w> <h> <conf>` after NMS
- `data/detections/frame_XXXX_vis.png` â€” input frame with boxes drawn

## Training a real model

The default `random_weights.py` makes the pipeline run but the
predictions are noise. To train:

```bash
pip install torch
python python/gen_dataset.py -n 2000      # synthetic shapes dataset
python python/train.py --epochs 30        # writes data/model.pt
python python/quantize.py                 # exports Q4.12 .mif weights
./scripts/run_sim.sh --trained
```

`python/model.py` defines the PyTorch model with the exact architecture
the RTL implements; training is YOLOv1-style (coord MSE for cells with a
target, BCE on objectness for all cells with `Î»_noobj=0.5`, BCE on
class logits for target cells).

## Porting from file feed to a real camera

The interface to swap is `tiny_yolo_top.ext_*` â€” currently the testbench
streams Q4.12 pixels in via that port. Replace the testbench with:

1. A camera-receiver block (e.g. MIPI CSI-2 â†’ debayer â†’ resize to 32Ă—32
   â†’ normalize to Q4.12) driving `ext_we / ext_waddr / ext_wdata`.
2. An HDMI/VGA framebuffer that reads predictions from `ext_rdata`
   after each `done` pulse and overlays boxes via a sprite blender.

Neither the conv pipeline nor the memory layout needs to change.
